# MiniPas+ 课程设计改进点汇总

本文根据 `2026-编译原理-课程设计.pptx`、`一个简单语言的编译实例 (1).doc` 以及 `基础知识点` 文件夹内容，对当前 MiniPas+ 项目进行逐项核对。重点核对模块包括：词法分析、语法分析、中间代码生成、优化、目标代码生成。

## 0. 总体结论

当前项目已经具备课程设计前端系统的基本框架：

- Qt Widgets 可视化界面已经完成。
- `Lexer / Parser / TypeTable / SymbolTable / Interpreter / CompilerFacade` 已经分层。
- 已支持 Token 序列、标识符表、常数表、符号表、record 类型表、array 类型表、四元式和解释执行结果展示。
- 已支持 `record`、`array`、函数声明的基础解析和表项展示。

但对照课程设计 PPT 和 Word 示例，目前仍偏向“演示型前端”，还没有完全达到文档中“完整编译过程”的要求。主要缺口集中在：

- Token 编码体系不够贴合 `(k,i,c,p)` 的课程示例。
- 符号表系统还没有完全实现 `SYNBL / TYPEL / AINFL / RINFL / PFINFL / CONSL / LENL / 活动记录` 的内部关联。
- 语法分析支持了部分扩展语法，但语义检查、作用域、函数体处理还不完整。
- 四元式生成覆盖赋值和算术表达式，但未覆盖数组、函数、控制流。
- 没有优化模块。
- 没有目标代码生成模块。

## 1. 词法分析

### 文档要求核对

PPT 和 Word 示例要求词法分析阶段输出：

- Token 序列。
- 关键字表 K。
- 界符表 P。
- 标识符表 I。
- 常数表 C。
- Token 形式建议类似：`(k,1)`、`(i,2)`、`(c,1)`、`(p,4)`。
- 标识符规则：字母开头，后接字母或数字。
- 常数规则：整数、实数。
- 关键字至少包括：`program, var, integer, real, char, begin, end`。
- 界符至少包括：`, : ; := * / + - . ( )`。

### 当前实现情况

当前已完成：

- 能识别关键字、标识符、整数、实数、界符、运算符、错误字符。
- 能维护 `identifierTable` 和 `constantTable`。
- Token 中包含 `type / lexeme / line / column`。
- 支持关键字大小写不敏感。
- 已扩展识别 `type, record, array, of, function, boolean, [, ]`。

### 不足之处

- 当前 Token 输出是字符串类别，如 `KEYWORD / IDENTIFIER / CONSTANT`，不符合 Word 示例中的 `(k,1)、(i,1)、(c,1)、(p,1)` 编码形式。
- 没有单独维护并展示“关键字表 K”和“界符表 P”，目前只展示 Token 序列、标识符表、常数表。
- 常数表只保存字符串，没有保存常数类型和值，例如 `integer / real`。
- 当前不支持字符常量，如 `'a'`，但语言类型里有 `char`。
- 当前没有注释处理，例如 `{ ... }` 或 `// ...`。
- 当前错误 Token 只记录非法字符，缺少更细的词法错误类型，例如非法实数、未闭合字符常量等。
- `array[1..10]` 依赖两个 `.` Token 拼出 `..`，可以工作，但词法层没有把 `..` 作为独立界符，和 Pascal 风格不够一致。

### 修改方向

- 增加 `KeywordTable` 和 `DelimiterTable`，固定编号，输出与课程示例一致：
  - 关键字 Token：`(k, 编号)`
  - 标识符 Token：`(i, 编号)`
  - 常数 Token：`(c, 编号)`
  - 界符 Token：`(p, 编号)`
- 在 `Token` 中增加：
  - `int code`
  - `int value`
  - 或者新增 `TokenKindCode`，用于 GUI 展示课程格式。
- 常数表改成结构体：
  ```cpp
  struct ConstantEntry {
      std::string text;
      std::string type;
      double numberValue;
  };
  ```
- 支持字符常量：
  ```pascal
  ch := 'a'
  ```
- 把 `..` 作为数组界符加入界符表。
- GUI 增加“关键字表 / 界符表”展示页，或放入“标识符与常数表”页。

### 当前完成状态

已完成：

- 已增加固定关键字表 K 和界符表 P。
- Token 已增加 `code/value`，可显示为 `(k,1)`、`(i,1)`、`(c,1)`、`(p,1)`。
- 常数表已改为结构化 `ConstantEntry`，包含常数字面量、类型和值。
- 已支持字符常量，例如 `'x'`。
- 已支持 `{ ... }` 和 `// ...` 注释跳过。
- 已将 `..` 作为独立界符加入 P 表，用于 `ARRAY[1..10]`。
- GUI 的 Token 表已增加“编码”列。
- GUI 的“标识符与常数表”页已展示关键字表 K、界符表 P、标识符表 I、常数表 C。

仍可后续增强：

- 未闭合 `{ ... }` 注释目前会被跳过到文件末尾，后续可改为生成明确的 `ERROR Token`。
- 字符常量目前按单字符处理，后续可补转义字符支持。

## 2. 语法分析

### 文档要求核对

Word 示例的基础文法包括：

- `PROGRAM -> program id SUB_PROGRAM .`
- `SUB_PROGRAM -> VARIABLE COM_SENTENCE`
- `VARIABLE -> var ID_SEQUENCE : TYPE ;`
- `COM_SENTENCE -> begin SEN_SEQUENCE end`
- `EVA_SENTENCE -> id := EXPRESSION`
- 表达式包含 `+ - * /` 和括号。

PPT 还建议扩展：

- 复杂数据类型。
- `If` 语句。
- `While` 语句。
- 子程序。

符号表部分要求声明语句获取标识符语义，并填写符号表。

### 当前实现情况

当前已完成：

- 使用递归下降分析。
- 支持 `program/type/record/array/function/var/begin/end`。
- 支持 record 字段声明。
- 支持 array 类型声明。
- 支持函数声明、参数、局部变量的基础解析。
- 支持普通变量声明、record 变量声明、array 变量声明。
- 支持赋值语句。
- 支持 record 字段访问，如 `x.a`。
- 支持 `+ - * /` 表达式和括号。
- 使用未声明变量会报错。
- record 字段不存在会报错。

### 不足之处

- 当前函数只做“声明解析和表项展示”，没有完整函数语义：
  - 不支持函数调用。
  - 不支持返回值赋值，如 `f := 1`。
  - 不支持函数活动记录。
  - 函数体内生成的四元式会混入全局四元式序列，语义上不够严谨。
- 当前数组只支持类型声明和变量声明：
  - 不支持数组元素访问，如 `z[1]`。
  - 不支持数组元素赋值，如 `z[1].a := 3.14`。
  - 不支持数组越界检查。
- 当前作用域不完整：
  - 参数、局部变量、全局变量都放在同一个符号表中。
  - 同名局部变量和全局变量无法按层次区分。
  - 符号表没有 `level`、`scope`、`owner` 等字段。
- 当前类型检查不完整：
  - `integer`、`real`、`char`、`boolean` 混合赋值未检查。
  - `char` 和 `boolean` 尚不能作为表达式值正确参与语义分析。
  - record 整体赋值、数组整体赋值没有明确规则。
- 当前没有 `if` 和 `while`，PPT 中把它们列为可扩展方向。
- 当前错误恢复能力弱，一旦遇到错误直接停止，不能继续分析后续语句。

### 修改方向

- 给符号表项增加：
  ```cpp
  int level;
  std::string scope;
  std::string owner;
  ```
- 函数声明应进入 `PFINFL` 函数表，保存：
  - 参数个数。
  - 参数传递方式：值参 `vf`、变参 `vn`。
  - 返回类型。
  - 入口四元式位置。
  - 局部变量区长度。
- Parser 中增加：
  - `parseArrayAccess()`
  - `parseFunctionCall()`
  - `parseIfStmt()`
  - `parseWhileStmt()`
- 左值扩展为：
  ```bnf
  LEFT_VALUE -> id LEFT_TAIL
  LEFT_TAIL -> . id | [ EXPRESSION ] | [ EXPRESSION ] . id | ε
  ```
- 增加类型推导函数，让 `parseExpression()` 返回表达式结果名和类型：
  ```cpp
  struct ExprResult {
      std::string place;
      std::string type;
  };
  ```
- 增加错误恢复机制，例如遇到错误后同步到 `;`、`end`、`.`。

## 3. 中间代码生成

### 文档要求核对

Word 示例要求生成四元式：

- 程序开始：`(program, I1, _, _)`
- 赋值：`(:=, C1, _, I2)`
- 运算：`(*, C1, C2, t1)`、`(+, t1, I2, t2)`
- 程序结束：`(end, I1, _, _)`

PPT 要求中间代码生成阶段输出：

- 四元式序列。
- 符号表系统。

### 当前实现情况

当前已完成：

- 支持 `(program, test, _, _)`。
- 支持 `(end, test, _, _)`。
- 支持普通赋值四元式。
- 支持 record 字段赋值四元式。
- 支持 `+ - * /` 表达式四元式。
- 支持临时变量 `t1, t2, ...`。
- 临时变量会进入符号表。
- GUI 可以展示四元式序列。

### 不足之处

- 当前四元式直接使用名字字符串，如 `x.a`、`3.14`、`t1`，没有使用课程示例中的 `I1 / C1 / t1` 表项引用形式。
- 临时变量类型固定为 `"temporary"`，没有根据表达式推导真实类型。
- 临时变量没有分配活动记录地址，`address = -1`。
- 函数声明目前没有生成独立函数入口、出口四元式。
- 函数体四元式没有单独分区，可能和主程序四元式混在一起。
- 数组声明没有生成四元式。
- 数组元素访问没有四元式。
- 不支持条件跳转四元式，例如 `j<, j=, jmp`。
- 不支持 `if`、`while` 的控制流四元式。
- 不支持函数调用四元式，例如 `param`、`call`、`return`。

### 修改方向

- 统一四元式操作数表示方式：
  - 方案一：保持字符串，GUI 友好。
  - 方案二：增加引用形式，更贴近课程示例：
    ```cpp
    struct Operand {
        std::string table; // I, C, T
        int index;
        std::string displayName;
    };
    ```
- 临时变量应带类型和地址：
  ```cpp
  newTemp(type)
  ```
- 增加数组元素四元式，例如：
  ```text
  ([] , z, i, t1)
  (:= , 3.14, _, z[i].a)
  ```
- 增加函数相关四元式：
  ```text
  (func, f, _, _)
  (param, m, _, _)
  (call, f, 1, t1)
  (return, value, _, _)
  (endfunc, f, _, _)
  ```
- 增加控制流四元式：
  ```text
  (j<, a, b, L1)
  (jmp, _, _, L2)
  ```

## 4. 优化

### 文档要求核对

Word 示例中明确包含“优化”阶段：

原四元式：

```text
(*, C1, C2, t1)
(+, t1, I2, t2)
```

优化后：

```text
(+, 10, I2, t2)
```

这属于常量折叠和删除无用临时变量。

### 当前实现情况

当前没有独立优化模块。

### 不足之处

- 没有 `Optimizer` 类。
- 没有保存优化前/优化后四元式。
- 没有常量折叠。
- 没有常量传播。
- 没有公共子表达式消除。
- 没有无用临时变量删除。
- GUI 没有“优化后四元式”展示页。
- `CompileResult` 中没有 `optimizedQuadruples` 字段。

### 修改方向

- 新增：
  ```text
  include/Optimizer.h
  src/Optimizer.cpp
  ```
- 定义接口：
  ```cpp
  class Optimizer {
  public:
      std::vector<Quadruple> optimize(const std::vector<Quadruple>& input);
  };
  ```
- 第一阶段只做课程示例中最容易说明的优化：
  - 常量折叠：`3 * 4 -> 12`
  - 常量传播：`t1 = 12; 2 + t1 -> 2 + 12`
  - 删除只被用一次且已折叠的临时变量。
- `CompileResult` 增加：
  ```cpp
  std::vector<Quadruple> optimizedQuadruples;
  ```
- GUI 增加“优化后四元式”标签页，或在“四元式序列”页上下分区展示优化前后。

## 5. 目标代码生成

### 文档要求核对

Word 示例要求把四元式翻译为简单目标代码，例如：

```asm
example SEGMENT
DB 12 DUP(0)
ASSUME CS:example, DS:example
MOV AX,2
MOV [0000H],AX
MOV AX,10
ADD AX,[0000H]
MOV [0008H],AX
MOV AX,[0008H]
MOV [0004H],AX
INT 21H
RET
example ENDS
```

PPT 中也提到完成前端后，有能力者建议加入后端设计。

### 当前实现情况

当前没有真正目标代码生成。

当前的 `Interpreter` 是四元式解释执行器，只能执行：

- `:=`
- `+`
- `-`
- `*`
- `/`

它不是目标代码生成器。

### 不足之处

- 没有 `CodeGenerator` 类。
- 没有目标代码输出字段。
- 没有根据符号表地址生成 `[0000H]` 这类地址引用。
- 没有生成 `SEGMENT / ASSUME / DB / MOV / ADD / SUB / MUL / DIV / INT / RET / ENDS`。
- 没有处理 record 字段地址，例如 `x.a` 应转换为 `base(x) + offset(a)`。
- 没有处理 array 元素地址，例如 `z[i]` 应转换为 `base(z) + (i-low) * elementSize`。
- 没有处理函数活动记录、参数区、局部变量区、临时变量区。
- GUI 没有目标代码展示页。

### 修改方向

- 新增：
  ```text
  include/CodeGenerator.h
  src/CodeGenerator.cpp
  ```
- 定义接口：
  ```cpp
  class CodeGenerator {
  public:
      std::vector<std::string> generate(
          const std::string& programName,
          const std::vector<Symbol>& symbols,
          const std::vector<RecordType>& records,
          const std::vector<ArrayType>& arrays,
          const std::vector<Quadruple>& quadruples);
  };
  ```
- `CompileResult` 增加：
  ```cpp
  std::vector<std::string> targetCode;
  ```
- 第一阶段目标代码可以仿照 Word 示例，只支持：
  - `program`
  - `end`
  - `:=`
  - `+ - * /`
  - 普通变量地址
  - record 字段地址
- GUI 增加“目标代码”标签页。

## 6. 符号表系统专项改进

### 文档要求核对

PPT 第 26 页要求复杂数据结构和函数时的符号表系统包括：

- `SYNBL`：符号表主表。
- `TYPEL`：类型表。
- `AINFL`：数组表。
- `RINFL`：结构表。
- `PFINFL`：函数表。
- `CONSL`：常量表。
- `LENL`：长度表。
- 活动记录。

### 当前实现情况

GUI 已展示：

- `SYNBL总表`
- `TYPEL类型表`
- `RINFL结构表`
- `AINFL数组表`
- `LENL长度表`

当前缺失：

- `PFINFL` 函数表。
- `CONSL` 常量表作为符号表系统的一部分。
- 活动记录展示。
- 表之间的指针/索引关系。

### 修改方向

- 新增函数表结构：
  ```cpp
  struct FunctionInfo {
      std::string name;
      std::string returnType;
      int paramCount;
      int entryQuad;
      int localSize;
      int tempSize;
  };
  ```
- 新增参数结构：
  ```cpp
  struct ParameterInfo {
      std::string functionName;
      std::string name;
      std::string type;
      std::string passMode; // vf 或 vn
      int offset;
      int size;
  };
  ```
- `SYNBL.TYPE` 不直接存字符串，而是能指向 `TYPEL` 项。
- `SYNBL.ADDR` 对变量指向活动记录偏移，对函数指向 `PFINFL`。
- GUI 增加：
  - `PFINFL函数表`
  - `CONSL常量表`
  - `活动记录`

## 7. 建议实施顺序

建议不要一次性重写，按下面顺序推进：

1. **整理词法输出格式**
   - 增加 K/P/I/C 表编号。
   - Token 显示改为课程示例风格。

2. **补强符号表系统**
   - 加 `PFINFL`、`CONSL`、活动记录。
   - 给 `SYNBL`、`TYPEL`、`AINFL`、`RINFL` 建立索引关系。

3. **完善语义检查**
   - 类型检查。
   - 作用域。
   - 函数参数。
   - array 下标。

4. **扩展四元式**
   - 数组元素访问。
   - 函数声明/调用。
   - if/while 控制流。

5. **实现优化模块**
   - 先做常量折叠和无用临时变量删除。

6. **实现目标代码生成**
   - 先按 Word 示例生成简单汇编式文本。
   - 再考虑 record、array、函数活动记录。

## 8. 当前项目最适合优先补的三项

如果以课程验收效果为目标，建议优先补：

1. **Token 编码输出和 K/P/I/C 表**
   - 这是 PPT 和 Word 示例最直观的检查点。

2. **优化后四元式展示**
   - 实现成本低，展示效果明显。
   - 可以直接复现 `2 * 5 -> 10` 的课程示例。

3. **目标代码文本生成**
   - 即使只支持普通变量和算术赋值，也能明显体现“完整编译过程”。
